Pre-Phase — Current State
Full Dashboard Audit
Every weakness identified across design, contrast, layout, and analytics intelligence.
Critical Issues
Critical Dark Team Colors — Invisible Nodes

Aston Villa (#670e36 maroon), Tottenham (#001c58 navy), Juventus (#2b2b2b), Newcastle (#241f20) and many others are nearly black. Player nodes, bar chart bars, and xG lines for these teams become indistinguishable from the #0d1117 background.

Fix: Add a get_effective_color(hex) function that computes relative luminance (WCAG formula). If luminance < 0.08, return a lightened variant (+40% L in HSL) or apply a bright outline/glow to all elements drawn in that color.

Critical VAEP Bar Labels on Dark Bars

The VAEP panel draws white text inside bar fills. For dark-colored teams (Aston Villa maroon), the bar is nearly invisible and the white label floats on near-black — unreadable without contrast checks.

Fix: For bars with luminance < 0.08, draw labels outside the bar (right-aligned after the bar end) instead of centered inside it. Use the team's effective_color for bar fill, not raw brand color.

Critical Pass Network Node Visibility

Player nodes inherit raw team color as their fill. Dark-colored teams produce invisible filled circles on the dark green pitch background. The gold playmaker ring is the only visible element.

Fix: Always draw nodes with a white or light-grey stroke of 1.8px minimum. For dark teams, set node fill to a semi-transparent lightened version and render a glow (path_effects withStroke, foreground='white', linewidth=3) on the scatter marker.

High-Priority Issues
High xG Flow Line Overlap

When both teams have similar cumulative xG values (e.g. 0.4 vs 0.5), the two lines visually stack. The area fills overlap and create a muddy brown zone that conveys nothing.

Fix: Reduce fill alpha to 0.06 (from 0.10). Add a subtle step-chart option: draw xG as steps at each shot minute, not a smooth slope — this communicates "bursts of danger" more honestly.

High Possession Origin Arrow Clutter

When both teams have 10+ shots each, 20+ arrows are drawn across the same half-pitch. The result is an illegible scribble, especially when origins and endpoints cluster in similar zones.

Fix: Cap displayed arrows at 8 per team (highest xG shots only). Add arrow alpha proportional to shot xG value (low-xG shots = very faint). Separate the two team's arrows into distinct left/right pitch halves.

High Counterattack Timeline Label Soup

Even at 30-event cap, when events cluster in the first 30 minutes (common in high-press games), minute labels ("12'", "13'", "14'") overlap and become unreadable. The xG dot sizes also overlap.

Fix: Use a stepped y-axis for labels (alternate top/bottom at +0.15 / -0.15 offsets). Apply minimum 3-minute spacing rule — only label events 3+ minutes apart. For denser periods, show only the highest-xG event.

High Score Banner — Congested Scorers List

Games with 4+ goals (like Man City 4-0) have scorer lists that overflow the banner height and text clips. The minute annotations (47', 49', 59', 63') can push beyond the allocated banner area.

Fix: Calculate banner height dynamically: base_h = 0.08, add 0.012 per goal beyond 2. Or collapse all scorers into a single comma-separated string if count > 3.

High Shot Map — Home/Away Directionality

Both teams' shots are mirrored to attack right, but the "away attacks ←" / "home attacks →" text labels are placed statically. When team names are long, these labels clip into the shot clusters.

Fix: Move directional labels to the very top of the pitch (y=78) as small italics with background bbox. Use abbreviated team names (last word only) in labels to prevent overflow.

High Background Too Dark — No Breathing Room

The #0d1117 background is GitHub-dark. Panel backgrounds (#13181f) are only 6 points lighter. This creates zero visual hierarchy between the page background, cards, and pitch areas — everything reads at the same depth level.

Fix: Shift BG to #0f1117, BG_PANEL to #161d27, BG_CARD to #1a2235. This 3-tier separation creates proper depth without losing the dark aesthetic.

Medium Issues — Conceptual Gaps
Medium VAEP Has No Role Tags

The top-5 VAEP panel shows bar lengths but gives no context about how the player contributes. A defender with high VAEP from interceptions looks identical to a striker with VAEP from shots.

Fix: Add a "Role" column based on action type mix: if >60% of VAEP comes from "Shot" events → "Finisher 🎯". If >60% from "Pass" → "Progressor". If >50% from "Dribble"/"Carry" → "Carrier". Render as small colored tags next to player name.

Medium xG Flow Has No Momentum Context

The cumulative xG chart shows total build-up but not momentum swings. A team can create 0.8 xG in 5 minutes (dominant) or 0.8 xG across 60 minutes (passive) — and both look like the same slope.

Fix: Add a "Pressure Index" area chart behind the xG lines: a rolling 10-minute window of final-third entries (loc_x > 80). Plot as a faint area fill behind the main xG step chart to show territorial dominance.

Medium Network Is "Soupy" in High-Pass Games

The current MIN_PASSES=3 threshold still produces extremely dense graphs in possession-heavy games (Man City, Barcelona). With 15+ active players and dozens of valid edges, the network becomes visually impenetrable.

Fix: Make threshold dynamic: MIN_PASSES = max(3, int(total_edges * 0.08)). This auto-scales to show the top 8% most frequent connections only. Also limit max displayed edges to 20 total.

Medium Insight Strip Is Too Generic

The 3 auto-generated bullets ("Manchester City created 3.81 more xG...") are always in the same format regardless of what actually happened tactically. They read like template text.

Fix: Add 4 more conditional bullet types: pressing intensity (pressure events per defensive action), set piece dependency (% of shots from set pieces), transition speed (avg seconds from turnover to shot), and defensive line height (avg defensive line x-position).

Medium Game State Table Is Hard to Parse

The "Before vs After 47' Goal" table uses small delta text underneath main values. The three columns (Before → After with delta) create too much visual noise for a static image at small font sizes.

Fix: Replace with a horizontal sparkline-style comparison: a small bar showing "before" as grey and "after" as team color, with just the delta (↑8% or ↓3%) as the only text. Far more scannable at a glance.

Medium No Progressive Carry Visualization

The possession origin arrows only show shot origins, not how the ball was physically carried into dangerous areas. Progressive carries (moving the ball 10+ yards toward goal) are invisible in the current dashboard.

Fix: Optionally add a "top 5 progressive carries per team" overlay on the tactical network panel — thin arrows in a complementary color (teal for home, orange for away) showing the longest carries into the final third.

Low-Priority Polish
LowFont Sizes Inconsistent

Axis labels use 7px, 8px, and 8.5px across different panels with no clear hierarchy system.

Fix: Standardize: panel titles=11px, axis labels=8px, data labels=7px, captions=7px italic. Apply globally via rcParams update.

LowNo Defensive Shape

The network only shows passes. Defensive positioning (average location when out of possession) is not shown.

Fix: Add an optional second network layer — semi-transparent colored zones showing avg defensive position using pressure events.

LowPlaymaker Annotation Overlaps

The playmaker callout box uses fixed offset logic that can clip into the pitch boundary for wingers near x=110 or x=10.

Fix: Add pitch-boundary clamp: ox = np.clip(ox, 12, 108), oy = np.clip(oy, 8, 72). Already partially done but needs tightening.

LowLegend Is Too Small

Network legend uses fontsize=5.2, which is below readable threshold on most displays at 160 DPI.

Fix: Increase to fontsize=6.5 minimum. Reduce number of legend entries to 4 (remove Sub entry, note in caption instead).

Task 1.1 — Luminance-Based Color System
01
Implement get_effective_color() with WCAG Luminance
Add this function immediately after the TEAM_COLORS dict. It computes relative luminance and, if below threshold, shifts the color to be viable on dark backgrounds. Also produces a "glow" foreground for stroke effects.
Python — Design Tokens · statsbomb_v9.py
# ── Luminance-safe color system ────────────────────────────── import colorsys def hex_to_rgb(h): h = h.lstrip('#') return tuple(int(h[i:i+2], 16)/255 for i in (0, 2, 4)) def relative_luminance(hex_color): rgb = hex_to_rgb(hex_color) def chan(c): return c/12.92 if c <= 0.04045 else ((c+0.055)/1.055)**2.4 r, g, b = [chan(c) for c in rgb] return 0.2126*r + 0.7152*g + 0.0722*b def get_effective_color(hex_color, threshold=0.06): """Return (display_color, stroke_color, is_dark).""" lum = relative_luminance(hex_color) if lum < threshold: # Shift to lightened HSL variant (+30% L) r, g, b = hex_to_rgb(hex_color) h, l, s = colorsys.rgb_to_hls(r, g, b) l2 = min(l + 0.35, 0.85) r2,g2,b2 = colorsys.hls_to_rgb(h, l2, s) light = '#{:02x}{:02x}{:02x}'.format(int(r2*255),int(g2*255),int(b2*255)) return light, '#ffffff', True return hex_color, '#000000', False # Usage in build_dashboard(): # raw_a = get_team_color(home); TEAM_A, TEAM_A_STROKE, A_DARK = get_effective_color(raw_a) # Pass A_DARK flag into all drawing functions to trigger label placement outside bars.

import pandas as pd import numpy as np def compute_elo(df, k=32, base=1500): """Compute rolling ELO from match results.""" ratings = {} elo_home, elo_away = [], [] for _, row in df.sort_values('date').iterrows(): h, a = row['home_team'], row['away_team'] rh = ratings.get(h, base); ra = ratings.get(a, base) elo_home.append(rh); elo_away.append(ra) exp_h = 1 / (1 + 10**((ra - rh) / 400)) result = 1 if row['home_goals'] > row['away_goals'] else \ 0.5 if row['home_goals'] == row['away_goals'] else 0 ratings[h] = rh + k * (result - exp_h) ratings[a] = ra + k * ((1-result) - (1-exp_h)) df['elo_home'] = elo_home df['elo_away'] = elo_away df['elo_diff'] = df['elo_home'] - df['elo_away'] return df def rolling_team_features(df, team_col, goal_col, n=5): """Rolling goals scored/conceded, win%, form streak per team.""" df = df.sort_values('date').copy() features = {} for team in df[team_col].unique(): mask = df[team_col] == team sub = df[mask].copy() sub['scored_roll'] = sub[goal_col].shift(1).rolling(n, min_periods=1).mean() sub['win_roll'] = (sub['result'].shift(1) == 'W').rolling(n, min_periods=1).mean() sub['streak'] = sub['result'].shift(1).map({'W':1,'D':0,'L':-1}).fillna(0) features[team] = sub[['date','scored_roll','win_roll','streak']] return features # Target variable: 0=Away win, 1=Draw, 2=Home win (3-class) # Or split into two binary models: home_win_prob + draw_prob

Recommended Bridge Strategy
Run the GH Actions workflow on a weekly schedule (cron: '0 6 * * 1') in addition to push triggers. This pulls the latest xgabora data, regenerates predictions for the week's upcoming fixtures, and rebuilds the site — keeping the Forecast tab fresh without any backend.

Phase 3 — Prompt to AI Coder
Build the FutDash prediction pipeline. Two parts: PART A — Python (scripts/): 1. feature_engineering.py: Load xgabora CSV files (one per league). Compute per-match features: ELO ratings (rolling, K=32), rolling 5-game goals scored/conceded, win%, form streak, home advantage flag, ELO diff, head-to-head win rate, league position difference. Output: features_df with columns matching the blueprint schema. 2. train_model.py: Train LightGBM 3-class classifier (H/D/A). Time-split: train pre-2023, validate 2023-2024. Log accuracy + Brier score. Save model with joblib to scripts/model.joblib. 3. generate_predictions.py: Run model on all matches. Output /web/public/data/predictions/{league}/{season}.json per blueprint schema. Flag "is_upset" when actual outcome had predicted prob < 0.30. Flag "has_statsbomb" + statsbomb_match_id when the match exists in open-data. PART B — React components (web/src/components/prediction/): 1. PredictionCard.tsx: Shows home/away team names + colors, date. Three ProbabilityBars (Home Win / Draw / Away Win) — animated fill on mount. For past games: actual outcome badge + upset tag. For StatsBomb games: "View Match Analysis" button → navigate to /match/:id. 2. ProbabilityBars.tsx: Three horizontal bars (green/grey/red). Width = probability %. Value shown as "41%" label at bar end. Animate from 0% to final value over 600ms on mount. 3. AccuracyTracker.tsx: Header component for PredictionsView. Shows "Last 20 predictions: X correct (Y%)". Fetches from current loaded data. 4. ForecastView.tsx: Filter by league dropdown. Shows upcoming match cards (actual=null). Groups by date. Shows model confidence label: "High confidence" if max prob > 0.55, else "Contested". All styling: Tailwind dark theme matching the FutDash palette.
